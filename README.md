# Сервис управления заявками

Небольшой CRUD-сервис для управления заявками (тикетами), сделанный в рамках
тестового задания. Бэкенд — **FastAPI + SQLite**, фронтенд — **React + TypeScript (Vite)**.

## Возможности

- Создание заявок (название, описание, приоритет).
- Список заявок с **фильтрацией** (по статусу, приоритету), **полнотекстовым поиском**
  (по названию и описанию), **сортировкой** (по дате создания/обновления, по приоритету
  с учётом важности) и **пагинацией** — всё выполняется на стороне сервера в SQL.
- Смена статуса заявки с соблюдением бизнес-правил.
- Удаление заявок — **только для администратора** (JWT-аутентификация).
- Понятные состояния загрузки / ошибки / пустого списка на фронтенде.

## Бизнес-правила

| Правило | Результат |
|---------|-----------|
| `done` — **терминальный** статус, его нельзя изменить или откатить | `409 Conflict` |
| Заявку в статусе `done` нельзя редактировать | `409 Conflict` |
| Разрешённые переходы: `new ⇄ in_progress → done`, `new → done` | иначе `409` |
| Удаление требует валидного токена администратора | `403 Forbidden`, если токен отсутствует/неверен |
| Заявка не найдена | `404 Not Found` |
| Невалидное тело запроса / параметры | `422 Unprocessable Entity` (FastAPI) |

Правила находятся в [`app/services/ticket_service.py`](backend/app/services/ticket_service.py)
в виде обычных функций, выбрасывающих доменные исключения, которые сопоставляются с
HTTP-кодами обработчиками в [`app/main.py`](backend/app/main.py) — поэтому они покрыты
юнит-тестами без HTTP-слоя.

## API

| Метод | Путь | Авторизация | Примечание |
|-------|------|-------------|------------|
| `POST` | `/auth/login` | — | `{username, password}` → `{access_token, token_type}` |
| `GET`  | `/tickets` | — | query: `status`, `priority`, `search`, `sort_by`, `order`, `page`, `page_size` |
| `POST` | `/tickets` | — | создание → `201` |
| `GET`  | `/tickets/{id}` | — | `404`, если не найдена |
| `PATCH`| `/tickets/{id}` | — | редактирование названия/описания/приоритета (блокируется при `done`) |
| `PATCH`| `/tickets/{id}/status` | — | смена статуса |
| `DELETE`| `/tickets/{id}` | **админ** | → `204` |

Интерактивная документация API (Swagger) доступна по адресу **http://localhost:8000/docs**.

## Быстрый старт (Docker)

```bash
docker-compose up --build
```

- Фронтенд → http://localhost:5173
- Бэкенд / Swagger → http://localhost:8000/docs

## Локальная разработка

### Бэкенд

```bash
cd backend
python -m venv .venv && source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt
uvicorn app.main:app --reload
```

Запускается на http://localhost:8000. Тесты:

```bash
cd backend
pytest
```

### Фронтенд

```bash
cd frontend
npm install
npm run dev
```

Запускается на http://localhost:5173 и проксирует `/api/*` на бэкенд (см.
[`vite.config.ts`](frontend/vite.config.ts)).

## Учётные данные администратора

```
логин:  admin
пароль: admin
```

Нажмите **«Вход для администратора»** в интерфейсе, чтобы авторизоваться; кнопка
удаления и админские элементы появляются только после входа. Учётные данные
настраиваются через переменные окружения (`ADMIN_USERNAME`, `ADMIN_PASSWORD`).

## Допущения и проектные решения

- **Статусы**: `new`, `in_progress`, `done`. `done` — терминальный.
- **Приоритеты**: `low`, `medium`, `high`; сортировка по приоритету выполняется по
  важности (а не по алфавиту) через SQL-выражение `CASE`.
- **Область авторизации**: аутентификация требуется только для удаления. Создание,
  просмотр списка и смена статуса доступны анонимно — это соответствует заданию
  (админ ограничивает именно *удаление*). Аутентификация — JWT: `POST /auth/login`
  возвращает bearer-токен, используемый для `DELETE`.
- **База данных**: SQLite через SQLAlchemy 2.x. В Docker файл БД хранится на
  именованном томе; локально создаётся как `backend/tickets.db` при первом запуске.
- **Фильтрация/сортировка/пагинация** всегда выполняются в базе данных, а не на клиенте.

## Структура проекта

```
backend/
  app/
    core/        # конфигурация + JWT/безопасность
    db/          # engine, сессия, Base
    models/      # SQLAlchemy-модель Ticket + перечисления
    schemas/     # Pydantic-схемы запросов/ответов
    services/    # бизнес-правила + запросы (ядро, покрытое тестами)
    api/         # маршруты + зависимость авторизации
    main.py      # приложение, CORS, обработчики исключений
  tests/         # pytest: бизнес-правила, эндпоинты, авторизация
frontend/
  src/
    api/         # fetch-клиент + обёртки эндпоинтов
    hooks/       # хуки TanStack Query, useAuth
    components/  # Table, Form, Filters, Pagination, LoginModal, StateViews
    types/       # общие TS-типы/перечисления
    App.tsx
docker-compose.yml
```
